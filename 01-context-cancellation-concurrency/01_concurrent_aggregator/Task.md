# Kata 01: Fail-Fast агрегатор данных

**Целевые идиомы:** управление конкурентностью (`errgroup`), передача `context`, паттерн Functional Options
**Сложность:** 🟡 Средняя

---

## 🧠 Зачем это нужно

В других языках для параллельного получения данных часто используют `Promise.all` или строгие пулы потоков. В Go разработчики обычно начинают с `sync.WaitGroup`, но быстро понимают, что ему не хватает двух критически важных вещей для продакшена: **передачи ошибок** и **отмены через Context**.

Если ты запускаешь 10 горутин и первая завершается с ошибкой, `WaitGroup` всё равно будет ждать, пока остальные 9 закончат работу.
**Идиоматичный Go работает по принципу fail fast — завершаться сразу при ошибке.**

---

## 🎯 Сценарий

Ты разрабатываешь **backend для пользовательского дашборда**.
Чтобы отобразить дашборд, нужно получить данные из двух независимых (моковых) микросервисов:

1. **Profile Service** (возвращает `"Name: Alice"`)
2. **Order Service** (возвращает `"Orders: 5"`)

Эти запросы нужно выполнять **параллельно**, чтобы уменьшить задержку.

Но если **любой из сервисов** возвращает ошибку или **истекает глобальный таймаут**, вся операция должна **немедленно прерваться**, чтобы не тратить ресурсы.

---

## 🛠 Задача

Создай структуру `UserAggregator` и метод `Aggregate(id int)`, который будет координировать получение этих данных.

---

## 1️⃣ Функциональные требования

* [ ] Агрегатор должен быть **настраиваемым** (таймаут, логгер) без огромного конструктора.
* [ ] Оба сервиса должны вызываться **параллельно**.
* [ ] Результат должен объединять оба ответа:

```
User: Alice | Orders: 5
```

---

## 2️⃣ Ограничения «идиоматичного Go» (критерии pass/fail)

Чтобы решить эту задачу правильно, **обязательно соблюдай эти правила**:

* [ ] **Нельзя использовать `sync.WaitGroup`**
  Нужно использовать `golang.org/x/sync/errgroup`.

* [ ] **Никакого "parameter soup"**
  Используй **паттерн Functional Options** в конструкторе
  (например `New(WithTimeout(2s))`).

* [ ] **Context — главный**
  `context.Context` должен быть **первым аргументом** во всех методах.

* [ ] **Очистка ресурсов**
  Если Profile Service падает с ошибкой, запрос к Order Service должен **сразу отмениться** через `context`.

* [ ] **Современное логирование**
  Используй `log/slog` для структурированных логов.

---

## 🧪 Самопроверка (протестируй себя)

Проверь код на этих сценариях.

### 1️⃣ «Медленный сервис»

* Установи таймаут агрегатора: `1s`
* Сделай мок одного сервиса с задержкой `2s`

**Условие успеха:**
функция должна вернуть ошибку

```
context deadline exceeded
```

ровно через **1 секунду**.

---

### 2️⃣ «Эффект домино»

* Profile Service сразу возвращает ошибку
* Order Service работает `10 секунд`

**Условие успеха:**
функция должна вернуть ошибку **немедленно**.

Если она ждёт 10 секунд — значит ты **не реализовала корректную отмену через context**.
